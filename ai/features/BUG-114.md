# Bug Fix: [BUG-114] AI Feature Safety & Rate Limiting

**Status:** queued | **Priority:** P2 | **Date:** 2026-02-16
**Bug Hunt Report:** [BUG-106](features/BUG-106-bughunt.md)

## Findings in This Group

| ID | Severity | Title |
|----|----------|-------|
| F-047 | high | No client-side rate limiting for AI receipt scanning -- cost spike risk |
| F-048 | medium | No rate limiting on AI cancellation features -- 3 separate LLM calls per action |
| F-049 | medium | AI nudge exposes full payment object to Edge Function -- PII leak risk |
| F-050 | high | Annual cost calculation missing biweekly/bimonthly periods -- wrong nudge amounts |

## Root Cause

Four related gaps in AI feature safety across receipt scanning, cancellation, and cooling-off:

1. **F-047:** The `ReceiptScanDialog` (`receipt-scan-dialog.tsx` line 106) has no client-side debounce or disable-on-pending guard on the "Scan" button beyond setting `state='scanning'`. While `scanMutation.mutateAsync` runs, a user can rapidly re-trigger scans by pasting new images, and nothing prevents queueing multiple mutations. The backend has rate limiting (10/hour via `checkRateLimit`), but a fast clicker can exhaust that budget instantly, each hit being a GPT-4V/Gemini vision call ($0.01-0.03 each).

2. **F-048:** The `CancellationDrawer` (`cancellation-drawer.tsx` lines 52-66) has three separate AI mutation triggers: `useCancellationScript()`, `useCancellationAlternatives()`, and `useCancellationGuide()`. All three hit the same Edge Function `cancellation-guide` which has ONE rate limit check (line 38). However, `useCancellationGuide` is a `useQuery` that fires automatically when the drawer opens (line 44: `enabled: !!paymentId`). The two mutations (`script`, `alternatives`) have no cooldown between clicks. Opening and closing the drawer repeatedly re-fires the guide query each time. All three share the same 10/hour rate limit pool, so spamming one burns budget for all.

3. **F-049:** The `CoolingOffDialog` sends `paymentData` to the Edge Function at line 68: `body: { action: 'cooling_off_nudge', payment: paymentData }`. The component receives a filtered subset (name, amount_cents, currency, period, period_days) from `payment-form-dialog.tsx` lines 86-92. HOWEVER: (a) The Edge Function `cancellation-guide/index.ts` line 50 types `body.payment` loosely -- any extra fields the client sends would pass through to `generateCoolingOffNudge`. (b) The `name` field itself is user-entered free text that could contain personal information and is sent directly to the LLM prompt in `cooling-off.ts` line 71. (c) Server-side `cooling-off.ts` `PaymentInput` interface (line 18-23) accepts whatever the client sends with no whitelist validation. The risk is lower than the finding states (full payment object is NOT sent) but the data flow lacks explicit sanitization.

4. **F-050:** The `annualMultipliers` map in `cooling-off-dialog.tsx` (lines 41-46) and `PERIOD_MULTIPLIERS` in `cooling-off.ts` (lines 25-31) are missing `daily` (365), `biweekly` (26), and `bimonthly` (6) entries. The fallback `?? 12` treats any unknown period as monthly, causing: a daily $5 subscription to show as $60/year (should be $1,825), biweekly $20 to show as $240/year (should be $520). These period types may come from future PERIODS expansion or from receipt scan AI results. The same gap exists in both client-side and server-side calculations.

## Fix Approach

### Step 1: Client-side rate limiting for receipt scan (F-047)

Add a debounce/cooldown mechanism to `ReceiptScanDialog`:
- Disable the "Scan" button while `scanMutation.isPending` is true (it currently only checks `state === 'scanning'` but the button is hidden in that state anyway -- the actual gap is re-triggering via paste or re-selecting a file while mutation is in-flight)
- Add an `isScanning` ref or use `scanMutation.isPending` to block `handleScan` re-entry
- Add a minimum cooldown (e.g., 3 seconds) after a successful or failed scan before allowing another
- Consider adding a visible "X scans remaining" counter using the 10/hour budget

### Step 2: Client-side cooldown for cancellation AI features (F-048)

Add per-action rate limiting in `CancellationDrawer`:
- Disable "Generate Script" button after first successful generation (it already conditionally shows the result, but clicking the copy button when no script exists triggers a new mutation -- line 53-54)
- Disable "Find Alternatives" button after first successful result (same pattern, line 63-64)
- For `useCancellationGuide` query: set `staleTime` to avoid re-fetching on drawer reopen (e.g., `staleTime: 5 * 60 * 1000`)
- Consider a shared in-memory counter across all three cancellation hooks to track total AI calls per session

### Step 3: Sanitize payment data sent to AI nudge (F-049)

Harden the data flow in both client and server:
- **Server-side (`cooling-off.ts`):** Explicitly destructure only the needed fields from the payment input: `{ name, amount_cents, currency, period }`. Sanitize `name` by stripping anything that looks like PII (email patterns, phone numbers, card numbers) before including in the LLM prompt. Add a max-length check on `name` (100 chars).
- **Edge Function (`cancellation-guide/index.ts`):** Validate and whitelist the `body.payment` fields at lines 101-103 before passing to `generateCoolingOffNudge`. Only allow: `name`, `amount_cents`, `currency`, `period`, `period_days`.
- **Client-side (`cooling-off-dialog.tsx`):** Already sends filtered data -- no change needed here, but add a TypeScript `satisfies` check to catch future regressions.

### Step 4: Add missing period multipliers (F-050)

Update BOTH multiplier maps to include all period types:
- **Client-side (`cooling-off-dialog.tsx` lines 41-46):** Add `daily: 365`, `biweekly: 26`, `bimonthly: 6` to `annualMultipliers` map. Update custom period logic to handle edge cases.
- **Server-side (`cooling-off.ts` lines 25-31):** Add `daily: 365`, `biweekly: 26`, `bimonthly: 6` to `PERIOD_MULTIPLIERS` map. Change `custom: 12` fallback to require explicit `period_days` (same logic as client-side: `365 / period_days`).
- Consider extracting the period multiplier map to a shared constant (similar to how PERIODS is shared in `lib/constants.ts`) to prevent drift between client and server.
