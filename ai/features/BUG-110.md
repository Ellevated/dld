## Root Cause

All seven findings share a common root cause: **inconsistent timezone handling and hardcoded iteration guards throughout the date/time layer**.

The codebase mixes three different date-to-string strategies:

1. **`toISOString().split('T')[0]`** -- converts to UTC, then extracts date portion. Used in `expandPayments` (line 165 of `src/lib/expand-payments.ts`), `payment-calendar.tsx` (lines 50, 111, 167), and `use-streak.ts` (lines 29, 32). This produces the WRONG local date for users in negative UTC offsets (e.g., UTC-5 at 11 PM local = next day in UTC).

2. **`format(date, 'yyyy-MM-dd')`** from date-fns -- uses local timezone. Used in `computeNextOccurrence` (lines 104, 116 of the same file). This is correct for local-date semantics.

3. **`new Date(dateString)`** without timezone suffix -- parses `'2026-02-01'` as UTC midnight, which is Jan 31 at 7 PM in UTC-5. Used in `calc-monthly.ts` (line 47).

Additionally, the `count < 5` guard in `calc-monthly.ts` line 65 was intended as a safety limit but caps daily subscriptions to 5 occurrences per month (should be ~30). The 365-iteration limit in `computeNextOccurrence` is insufficient for daily payments with very stale dates (a date 2 years behind needs 730 iterations). And the schema validation at `schema.ts` line 46 uses `>` instead of `>=`, blocking a legitimate edge case where end_date equals next_payment_date (the "last payment" scenario).

## Fix Approach

### 1. Standardize on local-timezone date formatting (F-023, F-024, F-025)

Replace ALL instances of `toISOString().split('T')[0]` with `format(date, 'yyyy-MM-dd')` from date-fns. This is a project-wide pattern fix affecting 4 files.

**In `src/lib/expand-payments.ts` line 165:**
- Change `current.toISOString().split('T')[0]` to `format(current, 'yyyy-MM-dd')`
- The `format` import already exists on line 20

**In `src/features/calendar/components/payment-calendar.tsx`:**
- Line 50 (DayCell `dateKey`): replace with `format(date, 'yyyy-MM-dd')`
- Line 111 (`todayKey`): replace with `format(new Date(), 'yyyy-MM-dd')`
- Line 167 (grid day key): replace with `format(day, 'yyyy-MM-dd')`
- Add `import { format } from 'date-fns'` at top

**In `src/hooks/api/use-streak.ts`:**
- Line 29 (`today`): replace with `format(new Date(), 'yyyy-MM-dd')`
- Lines 31-32 (`yesterdayStr`): replace with `format(yesterday, 'yyyy-MM-dd')`
- Add `import { format } from 'date-fns'`

### 2. Fix date parsing in calc-monthly.ts (F-023 variant)

**In `src/features/reports/utils/calc-monthly.ts` line 47:**
- Replace `new Date(payment.next_payment_date)` with local-timezone parsing
- Use `parseISO` from date-fns (which treats `'2026-02-01'` as local midnight, not UTC midnight)
- Alternatively, append `'T00:00:00'` to force local interpretation: `new Date(payment.next_payment_date + 'T00:00:00')`
- `parseISO` is preferred for consistency with `expand-payments.ts`

### 3. Remove count<5 guard in calc-monthly.ts (F-022)

**In `src/features/reports/utils/calc-monthly.ts` line 65:**
- The `count < 5` guard on the while loop caps daily subscriptions to 5 per month
- A daily subscription has ~30 occurrences per month; this guard severely undercounts
- Replace with a computed max based on period: `Math.ceil(daysInMonth / minPeriodDays) + 2`
- The `daysInMonth` can be computed from `monthEnd - monthStart` (already available)
- For daily payments (`period === 'custom'` with `period_days === 1`), max should be ~31
- For weekly, max should be ~5; for monthly, max should be ~1
- Use the same `minPeriodDays` logic as `expandPayments` uses for its `maxIterations` calculation

### 4. Increase iteration limit in computeNextOccurrence (F-021)

**In `src/lib/expand-payments.ts` line 109:**
- Change `iterations < 365` to `iterations < 3650`
- Rationale: daily payment 10 years behind needs 3650 iterations; the custom period_days max is 3650 (validated in schema), so a payment could be up to 10 years stale
- Also, the `while (current <= today)` uses `<=` meaning if `current === today`, it advances one more step. This is correct behavior per the test on line 26 of `expand-payments.test.ts` which confirms `computeNextOccurrence('2026-02-16', ...)` returns `'2026-02-16'` -- this works because the `>=` check on line 103 catches the today case before the loop. So F-021's "off-by-one on today" is actually handled. But the loop guard of 365 IS a real problem for daily/custom periods with very stale dates.
- Add a safety check: if `period` is unknown/invalid and doesn't advance (shiftForward returns same date), the `next <= current` break on line 111 handles it. No additional infinite-loop protection needed beyond increasing the limit.

### 5. Fix schema validation boundary (F-026)

**In `src/features/payments/data/schema.ts` line 46:**
- Change `return data.end_date > data.next_payment_date` to `return data.end_date >= data.next_payment_date`
- This allows setting end_date equal to next_payment_date (meaning "make this the last payment")
- Update the error message on line 51 to reflect the new rule: "Дата окончания должна быть не раньше даты следующего платежа"

**In `src/features/payments/data/schema.test.ts`:**
- Add a new test case: `it('accepts end_date equal to next_payment_date', ...)`
- Verify `paymentFormSchema.safeParse({ ...validPayment, end_date: new Date('2026-03-01') })` succeeds (same date as next_payment_date in test fixture)

### 6. Fix yearly payment backward navigation in expandPayments (F-027)

**In `src/lib/expand-payments.ts` lines 135-139 (backward navigation loop):**
- The backward shift for yearly payments uses `subYears(date, 1)`, stepping back one full year at a time
- For a short range (e.g., single month), after stepping back past `rangeEnd`, the forward loop (lines 141-145) may overshoot `rangeStart` entirely, missing the entry
- This happens because `shiftBack` for yearly goes back exactly 1 year, but `shiftForward` goes forward 1 year -- if the payment date doesn't fall within the range, it's skipped
- The fix: after the backward+forward positioning, if `current > rangeEnd`, the payment genuinely doesn't occur in this range. But the issue is that the backward loop stops at `current <= rangeEnd`, then forward jumps past. The real fix is to ensure backward goes far enough: overshoot backward by one extra period, then let forward find the right spot.
- Change backward loop condition from `while (current > rangeEnd)` to continue until `current <= rangeStart` (or `current <= rangeEnd` as is), then in the forward loop, advance from there. The current logic is correct for most cases. The specific yearly issue: if `next_payment_date` is e.g. `2027-03-15` and range is Feb 2026, backward goes to `2026-03-15` (within range), then the main loop starting at `current = 2026-03-15` is `<= rangeEnd` check at 2026-02-28 fails because Mar 15 > Feb 28. Entry is missed.
- Fix: after backward+forward positioning, if `current > rangeEnd`, bail out (no entry). But if `current >= rangeStart && current <= rangeEnd`, add it. The issue is that backward stops when `current <= rangeEnd` (Mar 15 for one-year range ending Dec 31), but for a Feb-only view, `rangeEnd` is Feb 28. The backward loop steps from 2027-03-15 to 2026-03-15, which is still > Feb 28, so it steps again to 2025-03-15, which is < rangeStart. Then forward goes to 2026-03-15, which is > rangeEnd(Feb 28), and the main loop doesn't enter.
- The actual fix: the backward loop should stop when `current <= rangeEnd`, not try to go below. For the range Feb 1-28 with yearly payment Mar 15: backward from 2027-03-15 steps to 2026-03-15 (> Feb 28), steps to 2025-03-15 (< Feb 28, stops). Forward from 2025-03-15 goes to 2026-03-15 (> Feb 28). Correctly, this payment does NOT occur in February, so there should be no entry. This is actually correct behavior.
- Re-examining: the real risk is with edge-crossing. A yearly payment on Jan 31 viewed in Jan: backward from 2027-01-31, stops at 2026-01-31 (which IS in Jan range). This works. The finding claims "yearly payments disappear from calendar when backward/forward alignment overshoots" -- this can happen if `shiftBack` from Feb 29 (leap) goes to Feb 28 (non-leap) then forward goes to Feb 29 (which doesn't exist, becomes Feb 28 or Mar 1 depending on date-fns). date-fns `subYears` handles this: `subYears(2028-02-29, 1) = 2027-02-28`. Then `addYears(2027-02-28, 1) = 2028-02-28`, not Feb 29. The original entry is lost.
- Fix approach for leap year issue: after positioning, compare computed date with the original `next_payment_date` day-of-month. If they differ due to month-end clamping, this is a known date-fns behavior. A practical fix: instead of using backward/forward from `next_payment_date`, generate entries by computing `next_payment_date + N*period` and `next_payment_date - N*period` directly from the original date, checking each against the range. This avoids the cumulative shift drift.

**Simplified fix for F-027:** Replace the backward/forward positioning with a direct approach:
1. Parse the base `next_payment_date`
2. Compute the nearest occurrence on or before `rangeEnd` by calculating how many periods fit between `next_payment_date` and `rangeEnd`, then subtracting whole periods
3. From that computed start, iterate forward through the range
4. This avoids cumulative drift from repeated shift operations on yearly boundaries

## Research Sources

- date-fns `parseISO` docs: parses ISO 8601 strings treating date-only strings as local dates (not UTC), unlike `new Date()` which parses them as UTC
- date-fns `format` docs: always uses local timezone, unlike `toISOString()` which converts to UTC
- MDN `Date.prototype.toISOString()`: "always returns the date in UTC timezone"
- MDN `new Date('YYYY-MM-DD')`: "date-only forms are interpreted as UTC time"
