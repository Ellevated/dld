# Bug Fix: [BUG-102] Protocol Bugs — Wrong Hook Output Format

**Status:** done | **Priority:** P0 | **Date:** 2026-02-16
**Bug Hunt Report:** BUG-101-bughunt.md

## Symptom
Three hooks produce output in the wrong protocol format:
1. `prompt-guard.mjs` uses PreToolUse protocol (`askTool`) in UserPromptSubmit hook — soft-block silently ignored
2. `session-end.mjs` uses UserPromptSubmit protocol in Stop hook — reminder never displayed
3. `session-end.mjs` has no try/catch — can crash (ADR-004 violation)

## Root Cause

### F-001: prompt-guard.mjs (P0)
`askTool()` outputs `{ hookSpecificOutput: { permissionDecision: 'ask' } }` — this is PreToolUse protocol.
UserPromptSubmit hooks expect `{ decision: 'block', reason: '...' }` or `{ decision: 'approve' }`.
Claude Code receives unrecognized JSON → treats as approve → the "consider /spark" suggestion is NEVER shown.

### F-002: session-end.mjs (P1)
Outputs `{ decision: 'approve', reason: '...' }` — this is UserPromptSubmit protocol.
Stop hooks likely don't support this output format → reminder about pending diary entries never displayed.

### F-003: session-end.mjs (P1)
Top-level code runs unprotected. If `process.cwd()` throws (deleted directory) or `stdout.write` fails (broken pipe), the hook crashes. All other hooks have try/catch per ADR-004.

### F-004: session-end.mjs (P2)
Doesn't import from `utils.mjs` — reinvents stdout writing, no `logHookError()` for diagnostics.

### F-005: validate-spec-complete.mjs (P2)
Lines 42-55 manually construct JSON instead of using `denyTool(reason)`. Duplicates protocol format, fragile, has redundant `process.exit(0)` after manual write (denyTool already exits).

## Fix Approach

### prompt-guard.mjs
Replace `askTool(message)` with `blockPrompt(message)` on line 57. The soft-block behavior becomes a hard block with an explanation — user sees the message and can retype with /spark. This is the correct UX since UserPromptSubmit has no "ask" mode.

### session-end.mjs
1. Add `import { logHookError, postContinue } from './utils.mjs'` (or suitable Stop helpers)
2. Wrap entire module in try/catch with `process.exit(0)` in catch
3. Replace manual stdout.write with appropriate helper from utils
4. If Stop hooks don't support output messages: write reminder to stderr, or simply exit silently and rely on other mechanisms for diary reminders

### validate-spec-complete.mjs
Replace manual JSON (lines 42-55) + redundant exit with `denyTool(reason)`. Remove `process.exit(0)` on line 56 (denyTool handles exit).

### Task 2: Rewrite session-end.mjs -- Add try/catch, utils import, correct Stop protocol

**Files:**
- Modify: `template/.claude/hooks/session-end.mjs` (full rewrite, 33 lines -> ~40 lines)

**Context:**
`session-end.mjs` has three issues: (1) no try/catch wrapper violating ADR-004, (2) doesn't import from utils.mjs so no `logHookError` diagnostics, (3) uses raw `process.stdout.write` without error handling. Research confirms Stop hooks support `{ decision: "approve", reason: "...", systemMessage: "..." }`. The fix wraps everything in a `main()` + try/catch pattern (matching all other hooks), imports `logHookError` and `readHookInput` from utils, and outputs the reminder via `systemMessage` on an approve decision so Claude sees it as context.

Note: Stop hooks have no dedicated helpers in utils.mjs (`approveStop`/`blockStop` don't exist). Rather than adding new helpers to utils.mjs (which is NOT in Allowed Files), we use the private `outputJson`-equivalent pattern: `process.stdout.write(JSON.stringify(...))` wrapped in try/catch, same as `outputJson` does internally. This is acceptable because Stop hooks are the only hook type without utils helpers, and adding helpers would require modifying utils.mjs which is out of scope.

**Step 1: Replace entire file content**

```javascript
// template/.claude/hooks/session-end.mjs
// COMPLETE REPLACEMENT

/**
 * Session End Hook - soft reminder about diary entries.
 * NEVER blocks session end, only adds context via systemMessage.
 *
 * Stop hook protocol: { decision: "approve"|"block", reason: "...", systemMessage: "..." }
 * We always approve (never prevent session end) but attach reminder as systemMessage.
 */

import { readFileSync, existsSync } from 'fs';
import { join } from 'path';
import { logHookError } from './utils.mjs';

function countPending(indexFile) {
  if (!existsSync(indexFile)) return 0;
  try {
    const content = readFileSync(indexFile, 'utf-8');
    return (content.match(/\| pending \|/g) || []).length;
  } catch {
    return 0;
  }
}

function main() {
  try {
    const projectDir = process.env.CLAUDE_PROJECT_DIR || process.cwd();
    const indexFile = join(projectDir, 'ai', 'diary', 'index.md');
    const pendingCount = countPending(indexFile);

    if (pendingCount > 5) {
      try {
        process.stdout.write(
          JSON.stringify({
            decision: 'approve',
            systemMessage: `Reminder: ${pendingCount} pending diary entries. Consider /reflect when convenient.`,
          }) + '\n',
        );
      } catch {
        // pipe closed — OK
      }
    }

    process.exit(0);
  } catch (e) {
    logHookError('session_end', e);
    process.exit(0); // fail-safe: never prevent session end
  }
}

main();
```

**Key changes from original:**
1. Added `import { logHookError } from './utils.mjs'`
2. Wrapped all code in `main()` function + try/catch (ADR-004)
3. Changed `reason` field to `systemMessage` (correct Stop protocol field for informational messages on approve)
4. Removed `reason` from approve decision (reason is for block explanations)
5. Inner try/catch around `stdout.write` for pipe-closed safety
6. `countPending()` now receives `indexFile` as parameter (testable)
7. Always calls `process.exit(0)` -- never blocks session end
