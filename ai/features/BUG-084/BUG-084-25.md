# Bug: [BUG-084-25] Sequential ID race on parallel spark invocations

**Status:** queued | **Priority:** P2 | **Date:** 2026-02-14
**Parent:** [BUG-084](BUG-084.md)

## Symptom
If two spark instances run in parallel (e.g., user triggers spark while autopilot is already running spark internally), both read the same max ID from backlog.md and create duplicate IDs.

Example:
- Spark A reads backlog: max = TECH-081
- Spark B reads backlog: max = TECH-081 (before A commits)
- Spark A creates BUG-082
- Spark B creates FTR-082
- Collision in backlog.md

## Root Cause
Spark completion.md lines 1-25 define ID determination protocol:
1. Read backlog (line 12)
2. Find max ID (line 14)
3. Add +1 (line 15)

But there's no file locking or atomic increment. Two concurrent sparks can both read the same max value before either commits.

This is a race condition inherent to file-based sequential numbering without coordination.

## Fix Approach
Add warning and operational guidance to completion.md. File-based atomicity is hard — document the limitation instead:

**After line 24 in completion.md, add new section:**

```markdown
### Race Condition Warning

**Sequential ID assignment has a race condition:**

If two spark instances run concurrently, they may:
1. Both read the same max ID from backlog.md
2. Both create specs with duplicate IDs (e.g., both create XXX-082)
3. Git merge conflict or backlog corruption

**Prevention:**
- Run spark from ONE terminal session at a time
- If autopilot is running, wait for it to finish before running spark manually
- Avoid parallel spark + autopilot workflows

**Detection:**
If you see duplicate IDs in backlog.md:
1. `grep "XXX-082" ai/backlog.md` — find the duplicates
2. Manually renumber the second occurrence
3. Update the corresponding spec filename
4. Recommit

**Future improvement:** File locking or atomic ID service (out of scope for v1).
```
