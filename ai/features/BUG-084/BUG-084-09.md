# Bug: [BUG-084-09] Worktree cleanup uses --force without checking uncommitted work

**Status:** queued | **Priority:** P1 | **Date:** 2026-02-14
**Parent:** [BUG-084](BUG-084.md)

## Symptom
In worktree cleanup flow:
- `template/.claude/skills/autopilot/finishing.md` line 41: `git worktree remove ".worktrees/{ID}" --force`
- `template/.claude/skills/autopilot/worktree-setup.md` line 96: `git worktree remove ".worktrees/{ID}" --force`

Both use `--force` flag unconditionally. From `worktree-setup.md` line 109:
```markdown
- ⛔ **NEVER** delete worktree with uncommitted changes
```

But the cleanup code violates this rule! `--force` destroys uncommitted changes without warning.

## Root Cause
Cleanup commands use `--force` for convenience (avoids "worktree has changes" error) but this bypasses safety checks.

## Fix Approach
Add `git status` check before `--force` removal:

**Replace:**
```bash
git worktree remove ".worktrees/{ID}" --force
```

**With:**
```bash
# Check for uncommitted changes
cd ".worktrees/{ID}"
if ! git diff-index --quiet HEAD --; then
  echo "ERROR: Worktree has uncommitted changes!"
  git status --short
  exit 1
fi
cd "$MAIN_REPO"

# Safe to force-remove (no uncommitted work)
git worktree remove ".worktrees/{ID}" --force
```

Alternatively, stash before removal:
```bash
cd ".worktrees/{ID}"
git stash push -m "autopilot-cleanup-{ID}"
cd "$MAIN_REPO"
git worktree remove ".worktrees/{ID}" --force
```

Prefer the error approach (fail-safe) over stash (might hide issues).

2. `template/.claude/skills/autopilot/worktree-setup.md` — line 96
