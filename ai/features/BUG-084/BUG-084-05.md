# Bug: [BUG-084-05] logHookError fails silently if ~/.cache not writable

**Status:** queued | **Priority:** P1 | **Date:** 2026-02-14
**Parent:** [BUG-084](BUG-084.md)

## Symptom
In `template/.claude/hooks/utils.mjs`, the `logHookError` function (lines 30-37) catches errors silently:

```javascript
export function logHookError(hookName, error) {
  try {
    const ts = new Date().toISOString();
    writeFileSync(getErrorLogPath(), `${ts} [${hookName}]: ${error}\n`, { flag: 'a' });
  } catch {
    // fail-safe: logging must never crash hook
  }
}
```

If `mkdirSync` fails in `getErrorLogPath()` (line 26) or `writeFileSync` fails, the error is swallowed with ZERO observability. Hook errors disappear into the void.

## Root Cause
The fail-safe pattern is correct (ADR-004: hooks must never crash), but it sacrifices ALL observability. If ~/.cache is not writable, users never know hooks are failing.

## Fix Approach
Add `console.error()` fallback before giving up:

```javascript
export function logHookError(hookName, error) {
  try {
    const ts = new Date().toISOString();
    writeFileSync(getErrorLogPath(), `${ts} [${hookName}]: ${error}\n`, { flag: 'a' });
  } catch (logErr) {
    // fail-safe: logging must never crash hook
    // but at least try console before giving up
    try {
      console.error(`[hook-error] ${hookName}: ${error}`);
    } catch {
      // truly hopeless, give up silently
    }
  }
}
```

This preserves fail-safe while adding minimal observability.
