## Framework Analysis

### TOC (Theory of Constraints)

**Core Constraint:** Absence of a verification/validation system for the template. The template is a production system (agents execute its instructions literally), but it is treated as documentation (no tests, no CI, no referential integrity checks).

**6 Causal Chains:**
| Chain | Root Cause | Findings | Injection Point |
|-------|-----------|----------|-----------------|
| A: Phantom References | Template-reality divergence, no validation | BH-001..006, BH-016 (7) | Reference validator script |
| B: Fail-Unsafe Hooks | ADR-004 principle without mechanism | BH-009, 010, 011, 029 (4) | try-catch on run-hook.mjs:43 |
| C: Regex/String Bypass | Naive matching without adversarial testing | BH-007, 012, 013, 014 (4) | Hook unit tests |
| D: Missing Atomicity | No transactional guarantees | BH-008, 018, 019, 020, 026 (5) | Write-then-verify pattern |
| E: Onboarding Dead-Ends | Expert-centric docs, no first-run testing | BH-015..017, 025, 028 (5) | Quick Start prerequisites |
| F: Broken Escalation | Decision trees without notification | BH-021, 022, 029 (3) | Notification mechanism |

**#1 Bottleneck:** BH-009 (run-hook.mjs:43 — one syntax error disables ALL safety hooks)

### TRIZ

**Key Technical Contradictions:**
- TC-1: Safety (fail-safe) vs Observability → **P35 Parameter Changes** (emit JSON, not filesystem)
- TC-2: Security (Allowed Files) vs Robustness → **P3 Local Quality** (null vs [] vs error)
- TC-3: Autonomy (autopilot) vs Data Integrity → **P23 Feedback** (verify after mutate)
- TC-5: Simplicity (template) vs Completeness → **P15 Dynamization** (conditional references)

**Key Physical Contradictions:**
- PC-1: Hooks must be silent (never crash) AND loud (report errors) → **Separation in space** (crash protection at runner, reporting at app level)
- PC-2: Allowed Files must be restrictive on error AND permissive when unspecified → **Separation on condition** (null vs [] vs error sentinel)
- PC-3: Template must be complete AND minimal → **Separation in time** (absent before bootstrap, present after)
