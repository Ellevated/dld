# Bug: [BUG-084-21] Council needs_human has no notification path

**Status:** queued | **Priority:** P2 | **Date:** 2026-02-14
**Parent:** [BUG-084](BUG-084.md)

## Symptom
Council can decide `needs_human` (line 124, 287, 343 in council/SKILL.md) but there's no mechanism to notify the user or pause autopilot execution. The decision exists in the output format but no handler ensures it reaches the human.

## Root Cause
Council SKILL.md defines `needs_human` as a valid decision status (line 124: `decision: approved | needs_changes | rejected | needs_human`), but:
1. No instruction exists to output a user-facing message when this status is chosen
2. Autopilot escalation.md (line 121: `needs_human → status: blocked`) mentions it but doesn't define notification mechanism
3. Missing procedural step: "set status=blocked → output clear message → pause execution"

## Fix Approach
Add explicit notification instruction to Council SKILL.md and clarify workflow in escalation.md:

1. In council/SKILL.md after line 343 (## After Council table), add new section:
```markdown
### When decision = needs_human

**CRITICAL:** Council must notify user and halt execution:

1. Set spec status to `blocked`
2. Add `## ACTION REQUIRED` section to spec:
   ```
   ## ACTION REQUIRED
   [Clear explanation of what Council needs from human]
   [Specific questions or decisions required]
   ```
3. Output to user:
   ```
   ⚠️ COUNCIL BLOCKED — Human decision required

   [Summary of the issue]
   [What Council needs from you]

   Spec marked as blocked: {spec_path}
   See ## ACTION REQUIRED section in spec.
   ```
4. Exit autopilot (do not continue to next task)
```

2. In escalation.md after line 121, clarify the notification path.
