# Bug: [BUG-084-06] extractAllowedFiles returns [] on read error → allow-all bypass

**Status:** queued | **Priority:** P1 | **Date:** 2026-02-14
**Parent:** [BUG-084](BUG-084.md)

## Symptom
In `template/.claude/hooks/utils.mjs`, the `extractAllowedFiles` function (lines 160-187) has this behavior:

```javascript
export function extractAllowedFiles(specPath) {
  try {
    const content = readFileSync(specPath, 'utf-8');
    // ... parse logic ...
    return allowed;
  } catch {
    return []; // fail-safe: missing spec = allow all
  }
}
```

The `isFileAllowed` function (lines 241-285) treats `[]` as "allow all":

```javascript
const allowedFiles = extractAllowedFiles(specPath);
if (allowedFiles.length === 0) {
  return { allowed: true, allowedFiles: [] }; // No Allowed Files section = allow all
}
```

**Problem:** Can't distinguish between:
1. "No spec file" (should allow all — user working outside autopilot)
2. "Spec file exists but read error" (should deny — something is broken)

If spec exists but has permissions error, file protection is bypassed.

## Root Cause
Single return value `[]` conflates two different states:
- No spec (legitimate allow-all)
- Spec read error (security failure)

## Fix Approach
Return `null` with error flag to distinguish states:

**extractAllowedFiles:**
```javascript
return { files: [], error: false };  // no spec, allow all
return { files: [...], error: false }; // spec parsed OK
return { files: [], error: true };   // read error, should deny
```

**isFileAllowed:**
```javascript
const result = extractAllowedFiles(specPath);
if (result.error) {
  return { allowed: false, allowedFiles: [], error: 'spec read failed' };
}
if (result.files.length === 0) {
  return { allowed: true, allowedFiles: [] }; // no spec = allow all
}
```

## Allowed Files
1. `template/.claude/hooks/utils.mjs` — lines 160-187 (extractAllowedFiles)
2. `template/.claude/hooks/utils.mjs` — lines 241-285 (isFileAllowed)
