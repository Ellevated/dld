**Status:** queued | **Priority:** P1 | **Date:** 2026-02-16
**Bug Hunt Report:** [BUG-106](features/BUG-106-bughunt.md)

## Findings in This Group

| ID | Severity | Title |
|----|----------|-------|
| F-028 | high | Silent fallback in currency `convert()` -- missing rates return unconverted amount |
| F-029 | critical | Tags chart double-counts payments with multiple tags -- pie chart sums to >100% |
| F-030 | critical | `monthlySum` uses `Math.round` after floating-point accumulation -- precision loss |
| F-031 | critical | `parseCents` regex allows multiple dots -- `'1.2.3'` parses to 120 cents silently |
| F-032 | high | `parseCents` silently truncates 3+ decimal places -- `'1.999'` becomes 199 cents |
| F-033 | critical | Currency rate input allows extreme values -- no bounds validation |

## Root Cause

The root cause across all six findings is **insufficient validation and precision handling in financial data paths**. Specifically:

1. **F-028**: `currency-store.ts` line 71-78 -- when `getRate()` returns `null` (missing rate), `convert()` silently returns the unconverted amount. A 100 USD payment appears as 100 RUB (~90x error). The `console.warn` only fires in DEV mode, so production users see silently wrong numbers.

2. **F-029**: `tags-chart.tsx` line 23-46 -- the `aggregateByTags` function iterates over each payment's tags in a `for...of` loop AND also handles the no-tag case, but for payments WITH tags, each tag gets the FULL `payment.amount_cents` added. A payment of 1000 cents with 3 tags adds 1000 to each tag's total (3000 total), inflating the pie chart to >100%.

3. **F-030**: `use-dashboard.ts` line 113-118 -- the `monthlySum` calculation multiplies cents (integer) by fractional factors (`4.33`, `1/3`, `1/12`) inside `reduce()`, accumulating floating-point rounding errors across all payments before a single final `Math.round()`. This violates ADR-001 (money in cents, no floats).

4. **F-031**: `money.ts` line 36 -- the regex `/^\d*\.?\d*$/` correctly rejects multiple dots. However, `split('.')` on line 40 only takes the first two parts: `const [whole, frac = ''] = normalized.split('.')`. If input is `'1.2.3'`, `split('.')` returns `['1', '2', '3']` but destructuring only takes `['1', '2']`, silently discarding `'3'`. The regex `\.?` means "zero or one dot" which DOES correctly reject `'1.2.3'`. **VERIFICATION**: Actually testing the regex `/^\d*\.?\d*$/` against `'1.2.3'` -- `\.?` allows zero or one dot, and `\d*` at the end only matches digits. So `'1.2.3'` would NOT match because after the optional dot and digits, there would be another `.3`. The regex IS correct. This finding may be a false positive. Need to verify at implementation time.

5. **F-032**: `money.ts` line 47 -- `(frac + '00').slice(0, 2)` takes only the first 2 characters of the fractional part. For `'1.999'`, `frac='999'`, padded = `'99900'`, sliced = `'99'`, so result is `199` cents. The third `9` is silently dropped. The existing test on line 34 of `money.test.ts` explicitly asserts this: `expect(parseCents('1.999')).toBe(199)`, meaning this was an intentional design choice. However, users entering `'1.999'` would expect either an error or `200` (rounded), not `199` (truncated).

6. **F-033**: `settings/currency/index.tsx` line 63-68 -- `saveEdit` only checks `val > 0` before calling `setOverride`. Values like `1e100`, `0.0000001`, or `999999999` are accepted, which breaks all currency conversions (a 100-cent payment converted at rate 1e100 overflows `Math.round`).

## Fix Approach

### F-028: Currency convert() silent fallback

1. Change `convert()` in `currency-store.ts` to return a result object or throw rather than silently returning the unconverted amount
2. Recommended approach: return `{ amount: number; converted: boolean }` or add a sentinel value. However, since `convert()` is called in many reduce operations, the simplest safe fix is:
   - Add a `convertSafe(amountCents, fromCurrency): { cents: number; isEstimate: boolean }` method that explicitly flags unconverted values
   - OR: keep the current signature but **log to a visible UI indicator** (a banner/badge) when any conversion falls back
   - The callers at `total-summary.tsx`, `card-summary.tsx`, `upcoming-widget.tsx`, `predictive-alert.tsx`, and `use-dashboard.ts` should all check whether conversion actually happened and display a warning (e.g., "some currencies could not be converted")
3. At minimum, move the `console.warn` outside the `DEV` guard so it always fires, and surface the warning to the UI

### F-029: Tags chart double-counting

1. In `tags-chart.tsx`, change `aggregateByTags` to **split payment amount evenly across its tags** rather than adding the full amount to each tag
2. For a payment of 1000 cents with 3 tags: add `Math.round(1000 / 3)` to each tag (with remainder handling to avoid rounding drift)
3. Alternative approach: count the payment once under its "primary" tag (first one). This is a product decision -- the spec recommends proportional splitting as the mathematically correct approach
4. Ensure the pie chart tooltip still shows meaningful absolute values

### F-030: monthlySum floating-point accumulation

1. In `use-dashboard.ts`, refactor the `monthlySum` calculation to keep everything in integer cents as long as possible
2. Instead of `convertedCents * factor` (which produces a float), multiply first then divide: e.g., for weekly: `convertedCents * 433 / 100` using integer arithmetic
3. Apply `Math.round()` per-payment rather than accumulating floats and rounding once at the end
4. Define `PERIOD_TO_MONTHLY_FACTOR` as rational fractions `{ numerator, denominator }` to avoid floating-point multiplication entirely:
   - weekly: `{ num: 433, den: 100 }` (4.33)
   - monthly: `{ num: 1, den: 1 }`
   - quarterly: `{ num: 1, den: 3 }`
   - yearly: `{ num: 1, den: 12 }`
   - custom: `{ num: 30, den: periodDays }`
5. Formula: `Math.round(convertedCents * num / den)` per payment, then sum the rounded integers

### F-031: parseCents multiple dots

1. **Verify** whether the regex actually rejects multi-dot input (it likely does based on analysis above)
2. If verification confirms the regex is correct, close this finding as false positive
3. If not, add an explicit early check: `if ((normalized.match(/\./g) ?? []).length > 1) throw`
4. Add test cases: `parseCents('1.2.3')` should throw `'invalid amount format'`

### F-032: parseCents silent truncation

1. Change truncation to either:
   - **Reject** inputs with more than 2 decimal places (throw an error) -- stricter but unambiguous
   - **Round** to nearest cent: `'1.999'` becomes 200 cents (banker's rounding or standard rounding)
2. Recommended: reject with a clear error message, because financial inputs should be explicit. The user likely made a typo
3. Update the existing test at `money.test.ts` line 33-34 which currently asserts truncation behavior
4. Add explicit Zod validation in `paymentFormSchema` to reject >2 decimal places at the form level, before `parseCents` is reached

### F-033: Currency rate bounds validation

1. In `settings/currency/index.tsx` `saveEdit()`, add bounds validation:
   - Minimum rate: `0.0001` (1/10000 -- covers all real-world currency pairs)
   - Maximum rate: `1_000_000` (covers extreme pairs like VND/BTC)
   - Show error toast if out of bounds
2. Also add validation in `currency-store.ts` `setOverride()` as defense-in-depth:
   - Reject `rate <= 0`, `rate > 1_000_000`, `!Number.isFinite(rate)`
3. Add `step` and `min`/`max` attributes to the rate `<Input>` in the settings UI
