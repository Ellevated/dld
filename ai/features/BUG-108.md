## Root Cause

The subscription upgrade and cancellation flows share a common architectural weakness: **absence of lifecycle guards for async operations that span user-dismissible UI boundaries**.

Specifically:

1. **Upgrade flow (F-008, F-009, F-011):** The `UpgradeDialog` component at `/Users/desperado/dev/plpilot/src/features/billing/components/upgrade-dialog.tsx` starts a 2-second polling interval (line 84) after `paymentResult === 'success'` and auto-closes the dialog when `subscription.plan !== 'free'` (line 106). The polling useEffect cleanup (line 97-101) correctly clears the interval on unmount, but the auto-close useEffect (line 105-116) has no dependency on the pollingRef and does NOT clear the timeout from line 95 on unmount. More critically, the auto-close fires based on subscription query cache, which may update BEFORE or AFTER the YooKassa webhook confirms the payment on the backend -- a 30-second timeout is insufficient for webhook delivery which can take 60-120 seconds under load. If the user closes the dialog manually while polling is active, the useEffect cleanup runs, but the `pollingRef` and interval are only cleaned up in the first useEffect -- a second open/close cycle can orphan the interval.

2. **SDK loading (F-010, F-012):** The `loadYooKassaSdk` function at `/Users/desperado/dev/plpilot/src/features/billing/lib/load-yookassa-sdk.ts` uses a module-level `loadPromise` variable (line 10) for deduplication. On failure, it sets `loadPromise = null` (line 41), which correctly allows retry. However, it does NOT remove the failed `<script>` tag from `document.head` (line 45 only appends, never removes). On retry, a second `<script>` tag is injected. If the SDK is blocked by an ad blocker, `window.YooMoneyCheckoutWidget` never appears, but the failed script tags accumulate. Furthermore, the `loadPromise = null` on line 36 (inside onload) means that after a successful load, a subsequent call checks `window.YooMoneyCheckoutWidget` first (line 22) -- but if the SDK script loaded but somehow failed to define the global (corrupted download), the function resolves successfully, then `useYooKassaWidget` at line 73 finds `!window.YooMoneyCheckoutWidget` and sets a local error, but the SDK is now in a poisoned state where `loadPromise` is null and `window.YooMoneyCheckoutWidget` is undefined, so every call re-injects a script.

3. **Cancel flow (F-013):** The `CancelDialog` component at `/Users/desperado/dev/plpilot/src/features/settings/billing/cancel-dialog.tsx` uses a single `cancelMutation` instance from `useCancelSubscription()` (line 28). All four action handlers (`handlePauseOneMonth`, `handlePauseThreeMonths`, `handleAcceptDiscount`, `handleFinalCancel`) share this mutation. While `cancelMutation.isPending` is used to disable buttons, it only covers THREE of the four handlers -- `handleAcceptDiscount` (line 75-79) has NO backend call at all, just a `toast.info` and close. This means the discount is never applied. Additionally, the `disabled={cancelMutation.isPending}` guard is per-step -- while step 1's pause button is disabled during pending, the user can still click "Continue cancellation" to reach step 3 and trigger `handleAcceptDiscount` or step 4 and trigger `handleFinalCancel` simultaneously, because step navigation is not guarded by `isPending`.

## Fix Approach

### 1. Extend polling timeout and add graceful degradation (F-008, F-009)

The subscription polling in `upgrade-dialog.tsx` needs three changes:

**a)** Increase the timeout from 30 seconds to 120 seconds (or make it configurable), matching the realistic upper bound for YooKassa webhook delivery.

**b)** When the timeout expires without subscription confirmation, do NOT auto-close the dialog. Instead, show a "delayed confirmation" message explaining the payment was received but activation is pending, with a "Check Status" button that manually triggers a single subscription query. This prevents the user from being left in limbo.

**c)** Ensure the `pollingRef` is properly cleaned up in ALL code paths: both the polling useEffect cleanup AND the auto-close useEffect cleanup AND the `handleClose` function must clear the interval. Currently `handleClose` (line 70-73) resets `confirmationToken` and closes, but does not clear `pollingRef.current`. Add `clearInterval(pollingRef.current); pollingRef.current = null;` to `handleClose`.

### 2. Clean up failed script tags and add retry guards (F-010, F-012)

In `load-yookassa-sdk.ts`:

**a)** Store a reference to the created `<script>` element. In the `onerror` handler, remove the script element from `document.head` before rejecting the promise. This prevents duplicate script tags on retry.

**b)** Add a `loadFailed` boolean flag (module-level, alongside `loadPromise`). Set it to `true` on error. On the next call, if `loadFailed` is true, reset it and proceed with a fresh attempt. This makes the state machine explicit: `idle -> loading -> (ready | failed) -> idle (on retry)`.

**c)** After a successful `onload`, verify that `window.YooMoneyCheckoutWidget` is actually defined before resolving. If the script loaded but the constructor is missing (corrupted download or polyfill failure), reject with a clear error message and remove the script tag. This prevents the "poisoned global" state.

### 3. Prevent widget orphaning during rapid mount/unmount (F-011)

In `use-yookassa-widget.ts`, the existing `cancelled` flag (line 67) and cleanup (lines 120-126) already handle the basic case. However, the `widgetRef.current = widget` assignment (line 99) happens BEFORE `widget.render()` (line 108) -- if the component unmounts between these two lines, the cleanup correctly calls `widget.destroy()` via widgetRef. But if the component unmounts DURING `widget.render()` (which is async), the render may complete after destroy is called.

**Fix:** Add a post-render cancelled check. After `await widget.render(containerId)` on line 108, check `if (cancelled)` again and call `widget.destroy()` if so. Also, set `widgetRef.current = null` in the cleanup function BEFORE calling destroy, to prevent double-destroy if React's StrictMode causes double-unmount.

### 4. Add mutual exclusion to cancel flow actions (F-013)

In `cancel-dialog.tsx`:

**a)** Add a local `isProcessing` state variable (`useState<boolean>(false)`) that is set to `true` at the start of ANY action handler and set to `false` in the finally block. Use this instead of `cancelMutation.isPending` for the `disabled` prop on ALL buttons and step navigation.

**b)** Fix `handleAcceptDiscount` (line 75-79) -- it currently has NO backend call. It must call `cancelMutation.mutateAsync({ action: 'accept_discount', reason, details })` or an equivalent API endpoint. If no backend endpoint exists yet for discount acceptance, the UI button should be removed or disabled until the endpoint is built. At minimum, add a TODO comment and make the function throw an error explaining the backend call is missing, rather than silently showing a fake success toast.

**c)** Disable ALL step-navigation buttons (Back, Continue, No thanks) while any action is pending. Currently `setStep(2)` on line 143, `setStep(3)` on line 194, and `setStep(4)` on line 252 are never disabled.

## Research Sources

- YooKassa webhook documentation indicates delivery can take up to 60-120 seconds under load or network delays, making a 30-second client-side timeout unreliable for payment confirmation.
- React useEffect cleanup pattern (React docs): cleanup runs synchronously on unmount but async operations initiated by the effect may still be in-flight. The `cancelled` flag pattern used in `use-yookassa-widget.ts` is the standard approach but must be checked after every `await`.
- DOM script injection best practice: always maintain a reference to injected script elements and remove them on failure to prevent DOM pollution and duplicate loading attempts.
