## Root Cause

All seven findings share a common root cause: **non-atomic state transitions in React component lifecycles**. Specifically:

1. **useEffect dependency arrays include mutable data sources** (F-014): The payment form's useEffect at `use-payment-form.ts:59` depends on `cards` (a TanStack Query result). Any background refetch of cards triggers a full form reset, destroying user edits mid-typing. The `cards` dependency exists solely for the scanData card-matching logic but fires on every cards query update.

2. **Refs used as one-shot flags without reset** (F-015): `fetchedRef` in `cooling-off-dialog.tsx:56` is set to `true` on first fetch but never reset to `false` when the dialog closes. Since CoolingOffDialog is rendered conditionally via `{pendingData && <CoolingOffDialog .../>}`, the component unmounts on close and remounts on next open -- which DOES reset the ref. However, if the parent keeps the component mounted (e.g., if `pendingData` is not cleared), the ref persists across dialog open/close cycles, showing Payment A's AI nudge for Payment B.

3. **Selective cache invalidation based on field-level heuristics** (F-016): `useUpdatePayment` in `use-payments.ts:162` conditionally invalidates dashboard vs. calendar queries based on which fields changed (`affectsFinancials` / `affectsSchedule`). This creates windows where dashboard shows one state and calendar shows another, because the payment data itself changed in both views but only one was refreshed.

4. **pendingData not cleared on cooloff dialog dismissal** (F-017): When user presses ESC or clicks outside the cooloff AlertDialog, `onOpenChange` calls `setShowCooloff(false)` but `pendingData` is never cleared. The form remains in a "submitted but not saved" limbo. If user edits form values and resubmits, old `pendingData` is overwritten, but if user simply closes the parent Sheet, `pendingData` persists in hook state.

5. **Two separate setState calls without batching guarantee** (F-018): In `data-table-row-actions.tsx`, clicking a row action calls `setCurrentRow(payment)` then `setOpen('delete')` as two sequential state updates. React 18+ batches these in event handlers, so this is safe in normal clicks. The real race is the `setTimeout(() => setCurrentRow(null), 500)` pattern -- if user clicks row A's delete, then within 500ms clicks row B's edit, the delayed `setCurrentRow(null)` from A's dialog close wipes row B.

6. **500ms setTimeout for currentRow cleanup** (F-019): Cards and tags dialogs both use `setTimeout(() => setCurrentRow(null), 500)` in three places each (onOpenChange for update, onOpenChange for delete, onSuccess for delete). This arbitrary delay creates a 500ms race window where the next user action can be corrupted by the previous cleanup.

7. **defaultValues computed once at mount, not synchronized** (F-020): Cards and tags mutate drawers pass `currentRow` to `useForm({ defaultValues: ... })`. React Hook Form only reads `defaultValues` on initial mount. If `currentRow` changes (user edits card A, closes, edits card B), the form still shows card A's data because the component key `card-update-${currentRow.id}` DOES change (forcing remount). However, the conditional rendering `{currentRow && <CardsMutateDrawer .../>}` means if `setCurrentRow(null)` from the 500ms timeout hasn't fired yet, the old component stays mounted with old data.

**Unifying principle:** All findings trace back to temporal coupling between React state updates, where the order and timing of state changes produces incorrect intermediate states visible to the user.

## Fix Approach

### Fix 1: Stabilize payment form reset (F-014)

Remove `cards` from the useEffect dependency array in `use-payment-form.ts`. The card matching for scanData should be computed lazily:
- Extract the scanData form reset into a separate effect that only runs when `scanData` changes and `open` transitions from false to true.
- Use a `hasInitializedRef` to track whether the form has been populated for this dialog session. Reset the ref when `open` becomes false.
- Compute `matchedCard` via `useMemo` instead of inside the effect, so it's available when the effect runs but does not trigger reruns.
- The dependency array should be `[open]` plus a stable ref for the initialization state, NOT `[currentRow, scanData, open, form, cards]`.

### Fix 2: Reset fetchedRef and nudge on dialog close (F-015)

In `cooling-off-dialog.tsx`:
- Add a cleanup effect that resets `fetchedRef.current = false` and `setNudge(null)` when `open` transitions to `false`.
- Structure: `useEffect(() => { if (!open) { fetchedRef.current = false; setNudge(null); } }, [open])`
- This ensures each dialog open gets a fresh AI nudge fetch for the current payment's data.

### Fix 3: Always invalidate all dependent queries on payment update (F-016)

In `use-payments.ts`, the `useUpdatePayment` mutation's `onSuccess`:
- Remove the `affectsFinancials` / `affectsSchedule` conditional logic.
- Always invalidate all payment-dependent queries: `payments.all`, `dashboard.all`, `calendar.all`, `reports.all`, `upcoming.all`, `sankey.all` (matching what `useCreatePayment` and `useDeletePayment` already do).
- Remove the `affectsFinancials()` and `affectsSchedule()` helper functions since they will be unused.
- Rationale: The performance benefit of selective invalidation is negligible (these are cheap Supabase queries), but the consistency cost is high (user sees conflicting data across views).

### Fix 4: Clear pendingData on cooloff dialog dismissal (F-017)

In `use-payment-form.ts`:
- When `showCooloff` is set to `false`, also clear `pendingData` to `null`.
- Wrap `setShowCooloff` in a handler that always pairs the two state updates: `const dismissCooloff = () => { setShowCooloff(false); setPendingData(null); }`.
- In `payment-form-dialog.tsx`, pass `dismissCooloff` as the `onOpenChange` for CoolingOffDialog instead of raw `setShowCooloff`.
- Also clear `pendingData` when the parent Sheet closes (in the `onOpenChange` handler).

### Fix 5: Atomic row selection with action-scoped snapshots (F-018)

In all three provider files (`payments-provider.tsx`, `cards-provider.tsx`, `tags-provider.tsx`):
- Change the pattern from two separate state updates (`setCurrentRow` + `setOpen`) to a single atomic action.
- Add a combined setter: `const openDialog = (type: DialogType, row: T) => { setCurrentRow(row); setOpen(type); }`.
- Expose `openDialog` from context instead of exposing `setCurrentRow` and `setOpen` separately to row-action components.
- Keep `setOpen(null)` for closing (which does not need row context).
- In row-action components (`data-table-row-actions.tsx` in payments, cards, tags), replace `setCurrentRow(x); setOpen('delete')` with `openDialog('delete', x)`.

### Fix 6: Replace setTimeout cleanup with dialog close callbacks (F-019)

In `cards-dialogs.tsx`, `tags-dialogs.tsx`, and `payments-dialogs.tsx`:
- Remove all `setTimeout(() => setCurrentRow(null), 500)` calls.
- Instead, use the `onOpenChange` callback properly: when the dialog/sheet transitions to closed state (`open` becomes false), set `currentRow` to null immediately. The 500ms delay was presumably added to allow close animations, but this should be handled by the dialog component's own animation lifecycle (Radix UI handles this internally via `onAnimationEnd`).
- For the delete success case, clear `currentRow` immediately in the `onSuccess` callback (which already happens -- the setTimeout is redundant since `setOpen(null)` is also called).
- If animation jank occurs from immediate cleanup, use `onAnimationEnd` on the dialog content rather than an arbitrary timeout.

### Fix 7: Force remount on currentRow change via key (F-020)

This is already partially handled by the `key={`card-update-${currentRow.id}`}` pattern in `cards-dialogs.tsx` and `tags-dialogs.tsx`. The real issue is the interaction with Fix 6: if `setCurrentRow(null)` fires immediately on close (no 500ms delay), then when user clicks "edit" on card B, `currentRow` changes from null to card B, which triggers a remount via the key change, and `defaultValues` are correctly read from card B.

Additionally, in `cards-mutate-drawer.tsx` and `tags-mutate-drawer.tsx`:
- Add a `useEffect` that calls `form.reset(...)` when `currentRow` changes while `open` is true, similar to how `use-payment-form.ts` handles it. This provides defense-in-depth if the key-based remount is somehow bypassed.
- Structure: `useEffect(() => { if (currentRow && open) { form.reset({ name: currentRow.name, ... }) } }, [currentRow, open])`

## Research Sources

- React Hook Form docs: `defaultValues` are only read on first render; use `reset()` to update after mount. The `key` prop on the component forces remount which re-reads defaultValues.
- TanStack Query docs: `invalidateQueries` marks queries as stale and triggers refetch. Selective invalidation can cause cache inconsistency when multiple views share underlying data.
- React 18 batching: State updates in event handlers are batched, but setTimeout callbacks are also batched in React 18 (unlike React 17). The race is not about batching but about the 500ms delay creating a window for new actions.
