# Bug Fix: [BUG-112] Authorization & Access Control

**Status:** queued | **Priority:** P0 | **Date:** 2026-02-16
**Bug Hunt Report:** [BUG-106](features/BUG-106-bughunt.md)

## Findings in This Group

| ID | Severity | Title |
|----|----------|-------|
| F-034 | critical | All queries rely 100% on RLS -- no client-side user_id filter on any data query |
| F-035 | critical | Payment update RPC allows updating any field -- no field-level authorization |
| F-036 | high | Receipt upload path not sanitized -- file extension from client MIME type |
| F-037 | medium | Subscription hooks use any-cast to bypass TypeScript -- zero type safety |
| F-038 | high | Unsanitized color values in cards/tags inline styles -- CSS injection risk |

## Root Cause

The application has a systemic absence of defense-in-depth across its authorization and input sanitization layers:

1. **RLS-only authorization (F-034):** Every Supabase query in `src/hooks/api/` uses `.from('table').select('*')` without any `.eq('user_id', ...)` filter. The application relies 100% on Supabase RLS policies to enforce row-level access. RLS is correctly configured (verified in migrations 00001, 00013, 00024), but if RLS is ever misconfigured, disabled during a migration, or bypassed via a SECURITY DEFINER function, all user data is exposed. There is zero client-side defense.

2. **RPC field whitelist missing (F-035):** The `update_payment_with_tags` RPC (migration 00024) accepts ALL payment fields as parameters and applies them via COALESCE. While `user_id` is not an RPC parameter (it's extracted from JWT), the function does have ownership verification. The original finding overstates the risk -- `user_id` cannot be changed through this RPC because it is not a parameter. However, there is still no field-level validation on the RPC inputs (e.g., `amount_cents` could be negative, `currency` could be arbitrary string, `period` could be invalid).

3. **Receipt path from MIME type (F-036):** In `use-payment-form.ts:133`, the file extension is derived from `scanImageFile.type` (client-provided MIME type). The path is `${paymentId}.${ext}`. While Supabase Storage enforces `allowed_mime_types` on the bucket (migration 00003), and the storage RLS requires the folder name to match the user's JWT sub (preventing path traversal to other users' folders), the upload path does NOT include a user folder prefix. The path is simply `${paymentId}.${ext}` -- there is no folder component, so the `foldername` RLS check would actually BLOCK this upload because `foldername(name)[1]` of `"some-uuid.jpg"` is not the user's ID. This means either (a) the RLS policy is not working as written, or (b) uploads silently fail. Additionally, the extension derivation has a fallback to `'jpg'` for any unknown MIME type, which means a manipulated MIME type would not cause path traversal but could result in mismatched content-type.

4. **any-cast on subscription hooks (F-037):** In `use-subscription.ts:22-25`, the Supabase client is cast to `unknown as UntypedClient` to bypass TypeScript type checking for tables not yet in `database.types.ts` (billing tables from migration 00022). This eliminates all column name, type, and structure validation at compile time for `user_subscriptions` and `billing_history` queries.

5. **Unsanitized color in inline styles (F-038):** Color values from `cards.color` and `tags.color` columns are rendered directly in JSX `style={{ backgroundColor: color }}` across 8+ files. While the form schema validates `^#[0-9A-Fa-f]{6}$` on the client-side Zod schema, the database column is `text` with no CHECK constraint. Any color value inserted directly via SQL, API manipulation, or migration would render unsanitized in the browser. React's style object prevents script injection via `style` attributes (React serializes CSS values safely), but a malicious value could still cause visual disruption.

## Fix Approach

### F-034: Add client-side user_id filter as defense-in-depth

1. Create a shared utility function `withUserFilter` that wraps Supabase queries and automatically adds `.eq('user_id', userId)` where applicable
2. Modify each query hook to include the user_id filter alongside RLS:
   - `usePayments()`, `usePaymentsList()`, `usePayment()` -- filter by user_id
   - `useCards()` -- filter by user_id
   - `useTags()` -- filter by user_id
   - `useReceipt()` -- filter by user_id (receipts table has user_id)
   - `useNotificationSettings()` -- filter by user_id
   - `useBotMode()` -- filter by user_id
   - `useSubscription()` -- filter by user_id
   - `useBillingHistory()` -- filter by user_id
   - `useStreak()` -- filter by user_id
   - `useCalendarPayments()` -- filter by user_id
   - `useReportPayments()` -- filter by user_id
   - `useSankeyData()` -- filter by user_id
   - `useExchangeRates()` -- NO user_id (shared table), skip
   - `useReferralSource()` -- filter by user_id
3. The user_id comes from Clerk's `useAuth()` hook (`userId` property)
4. For queries using `.single()`, the filter is especially important as it prevents PGRST116 errors from returning wrong-user data
5. This does NOT replace RLS -- it adds a second layer. Both must be present.

### F-035: Add field-level validation to RPC inputs

1. In the `update_payment_with_tags` SQL function, add CHECK constraints on input parameters:
   - `p_amount_cents` must be > 0 if provided
   - `p_currency` must be in the allowed enum if provided
   - `p_period` must be in the allowed enum if provided
   - `p_period_days` must be > 0 if provided and period is 'custom'
2. In the `insert_payment_with_tags` SQL function, add the same constraints
3. This validates at the database level, complementing the client-side Zod schema
4. Create a new migration file for these constraints

### F-036: Fix receipt upload path construction

1. In `use-payment-form.ts`, change the upload path to include user folder prefix: `${userId}/${paymentId}.${ext}`
2. This aligns with the existing Storage RLS policy that checks `(storage.foldername(name))[1] = user_id`
3. Validate the extension against an explicit allowlist (`['jpg', 'png', 'webp']`) instead of deriving from MIME type
4. If the MIME type does not match the allowlist, reject the upload rather than falling back to 'jpg'
5. Get `userId` from Clerk's `useAuth()` hook in the component

### F-037: Replace any-cast with proper types

1. Run `supabase gen types typescript` to regenerate `database.types.ts` with billing tables
2. If generation is not possible (migration 00022 not yet applied), manually add the `user_subscriptions` and `billing_history` table types to `database.types.ts`
3. Remove the `UntypedClient` type alias and `as unknown as UntypedClient` casts from `use-subscription.ts`
4. Use properly typed Supabase client for all billing queries
5. This restores compile-time column name and type checking

### F-038: Sanitize color values before rendering

1. Create a shared utility function `sanitizeColor(color: string | null): string | null` in `src/lib/color-utils.ts` that validates against `^#[0-9A-Fa-f]{6}$` regex
2. Apply this sanitizer at every render point where DB color values are used in inline styles (8 files identified)
3. Add a database CHECK constraint on `cards.color` and `tags.color` columns: `color IS NULL OR color ~ '^#[0-9A-Fa-f]{6}$'`
4. This provides three layers: form validation (Zod), render sanitization, and database constraint

## Research Sources

- Supabase RLS documentation -- RLS is first-line defense but not sufficient as sole authorization layer; client-side filters recommended as defense-in-depth
- React security model -- React's `style` attribute uses `CSSStyleDeclaration` which prevents script injection, but does not prevent visual disruption via arbitrary CSS values
- OWASP defense-in-depth principle -- never rely on a single layer of security
