## Root Cause

Eight UX dead-end bugs across billing, receipts, cancellation, and dashboard. The common theme is **incomplete error-state handling**: the code handles the "happy path" well but leaves users stranded when errors occur. Specific root causes:

1. **Technical error messages exposed to users** (F-039): `useYooKassaWidget` sets error to raw `err.message` (e.g., `"Element with id 'yookassa-payment-form' not found after 20 frames"`) which the `upgrade-dialog.tsx` renders verbatim.

2. **Warning toast for data loss** (F-040): `use-payment-form.ts:139,149` uses `toast.warning()` for receipt upload/link failures. Warnings auto-dismiss and are easy to miss. User loses receipt data without clear awareness.

3. **No back button on cancellation guide error** (F-041): `cancellation-drawer.tsx:265` renders a static error message with no retry button and no way to close/go back.

4. **Rate limit error without cooldown info** (F-042): `receipt-scan-dialog.tsx:121` shows "Подождите немного" but gives no indication of how long to wait.

5. **Polling timeout leaves spinner** (F-043): `upgrade-dialog.tsx:92-95` stops polling after 30s but does NOT update the UI. User sees infinite spinner.

6. **String-based rate limit detection** (F-044): `receipt-scan-dialog.tsx:120` checks `message.includes('Rate limit') || message.includes('429')`. Brittle — server error messages can change.

7. **Unsafe data URL parsing** (F-045): `receipt-scan-dialog.tsx:329` calls `result.split(',')[1]` assuming data URL format. If format differs, returns `undefined` sent to edge function.

8. **Alert fatigue from predictive alert** (F-046): `predictive-alert.tsx:33` shows alert when `urgentGroups.length === 0 && report.payment_count >= 2` — noise without actionable info.

## Fix Approach

### F-039: User-friendly widget error messages + retry capability
Map technical errors to user-friendly messages in `use-yookassa-widget.ts`:
- SDK load failure -> "Не удалось загрузить платёжную форму"
- Element not found -> "Платёжная форма не готова, попробуйте снова"
- Widget render error -> "Ошибка отображения формы оплаты"
- Generic -> "Произошла ошибка. Попробуйте обновить страницу"

### F-040: Escalate receipt failures from warning to persistent error
Change `toast.warning()` to `toast.error()` with `duration: Infinity` on lines 139 and 149.

### F-041: Add retry and close actions to cancellation guide error state
Replace static error `<div>` with error card containing retry button.

### F-042: Show cooldown timer on rate limit
Start local countdown timer (30 seconds), display "Слишком много запросов. Повторите через {N} сек.", disable scan button.

### F-043: Handle polling timeout with explicit UI state
Set `pollingTimedOut = true` on timeout, show message with manual refresh button.

### F-044: Replace string-based rate limit detection with status code check
Extract to helper function `isRateLimitError(err)`, use structured error code or status.

### F-045: Validate data URL format before splitting
Validate `reader.result` is string and contains comma before `split(',')[1]`.

### F-046: Fix predictive alert display logic
Change condition: `if (urgentGroups.length === 0) return null` — suppress when no urgent groups.
