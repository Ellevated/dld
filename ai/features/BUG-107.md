# Bug Fix: [BUG-107] Payment & Billing Financial Safety

**Status:** done | **Priority:** P0 | **Date:** 2026-02-16
**Bug Hunt Report:** [BUG-106](features/BUG-106-bughunt.md)

## Findings in This Group

| ID | Severity | Title |
|----|----------|-------|
| F-001 | critical | Payment token reused on retry -- double charge risk |
| F-002 | critical | Receipt upload silently fails after payment creation -- orphaned data |
| F-003 | critical | Discount acceptance has no backend call -- fake confirmation |
| F-004 | high | Refund eligibility off-by-one error on day 7 boundary + client-side time |
| F-005 | high | Refund toast hardcodes ruble symbol -- wrong currency for non-RUB users |
| F-006 | critical | Scan data maps one-time payments to monthly -- wrong recurring charge created |
| F-007 | medium | Refund dialog state resets on close -- allows duplicate refund attempts |

## Root Cause

These 7 findings share a common theme: **financial operations in the billing/payment pipeline lack transactional safety, proper idempotency, and correct data mapping**. The code was built for the happy path but omits critical guards that protect the user's money.

Specific root causes:

1. **F-001**: The `UpgradeDialog` "Try Again" button (`setConfirmationToken(null)`) clears the token, sending the user back to plan selection, but clicking "Pay" again calls `createCheckout()` which creates a NEW `create-payment` edge function call. The backend idempotency key includes an hour-bucket, so within the same hour it is safe, but across hour boundaries (or if the user changes plans), a second charge is created. The client-side has no guard to prevent calling `createCheckout` while a previous payment is still `pending` in YooKassa.

2. **F-002**: In `use-payment-form.ts:168-184`, `createPayment.mutate()` calls `uploadReceiptImage()` in `onSuccess`. If the upload fails (lines 138-141), a `toast.warning` fires but the payment is already committed. The receipt record insert can also fail independently (lines 148-149). This is a two-step non-atomic operation -- payment succeeds but receipt is orphaned.

3. **F-003**: In `cancel-dialog.tsx:75-79`, `handleAcceptDiscount()` shows `toast.info('Discount 25% will be applied')` and closes the dialog, but makes **zero API calls**. The `cancel-subscription` edge function supports cancel/pause/reactivate but has no `discount` action. The user believes they got a discount but will be charged full price.

4. **F-004**: In `use-subscription.ts:236`, `daysSince <= 7` uses `Date.now()` (client-side time). The backend (`request-refund/index.ts:53`) uses `daysSince > 7` (server-side). The client check is cosmetic (shows/hides the button) but uses client clock which can be wrong. More importantly, `<= 7` on client vs `> 7` on backend means day 7.0001 is eligible on client but rejected on server -- confusing UX.

5. **F-005**: In `refund-dialog.tsx:29`, the toast hardcodes the ruble symbol: `` `Vozvrat ${result.amount} ₽ oformlen` ``. But the backend returns `result.currency` (line 74 of `request-refund/index.ts`). While PLPilot currently only charges in RUB, the API returns the currency field and the display should use it.

6. **F-006**: In `use-payment-form.ts:80`, scan data with `period === 'one-time'` is silently mapped to `'monthly'`. A one-time payment (e.g., annual software license) becomes a monthly recurring charge. The user sees "monthly" in the form but may not notice the change.

7. **F-007**: In `refund-dialog.tsx:46-48`, closing the dialog resets step to `'initial'`, allowing the user to reopen and click "Refund" again. The backend has idempotency (YooKassa deduplicates refunds), but the UI allows a confusing double-submission. After a successful refund, the button should be disabled.

## Fix Approach

### F-001: Add client-side payment-in-progress guard

1. In `use-checkout.ts`, add a `paymentId` state that is set when `createCheckout()` succeeds (the backend returns `payment_id`).
2. If `paymentId` is already set and non-null, `createCheckout()` should skip the API call and return the existing `confirmationToken` from a stored ref, or require explicit clearing.
3. In `upgrade-dialog.tsx`, the "Try Again" flow after payment failure should call a new `resetCheckout()` that explicitly clears `paymentId` and `confirmationToken` together, documenting that this is intentional (user wants a new payment attempt).
4. Add `isRetrying` state to disable the "Pay" button while a previous payment is still pending.

### F-002: Make receipt upload failure non-silent and offer retry

1. In `use-payment-form.ts`, change `uploadReceiptImage()` to return a result indicating success/failure.
2. In the `onSuccess` callback of `createPayment.mutate()`, if receipt upload fails, store the payment ID and file reference in component state.
3. Show a persistent (non-auto-dismiss) toast or inline UI that says "Payment created, but receipt upload failed" with a "Retry Upload" button.
4. The retry button should call `uploadReceiptImage(storedPaymentId)` directly.
5. Consider wrapping the receipt insert + storage upload in the same try-catch with a rollback: if the DB insert fails, delete the storage file.

### F-003: Wire discount acceptance to a backend API call

1. In `cancel-dialog.tsx`, change `handleAcceptDiscount()` to call `cancelMutation.mutateAsync()` with a new action.
2. The `cancel-subscription` edge function needs a new action `'apply_discount'` that:
   - Creates a record in a `subscription_discounts` table (or updates `user_subscriptions` with a `discount_percent` and `discount_until` field)
   - Logs the offer acceptance in `cancellation_feedback`
3. The frontend handler should `await` the mutation and only show the success toast after confirmation.
4. Handle error state -- if the backend rejects (e.g., discount already applied), show an error toast.

### F-004: Align client and server refund eligibility logic

1. In `use-subscription.ts` `useIsRefundEligible()`, change the comparison from `daysSince <= 7` to `daysSince < 7` to match the backend's `daysSince > 7` boundary (both exclude exactly 7 days).
2. Better: change both to use `< 7` (client) and `>= 7` (server rejection) so that exactly day 7.0 is still eligible. Choose one consistent boundary and document it.
3. The client check is cosmetic (button visibility), so being slightly more generous than the server is acceptable -- but the day 7 edge case should be handled gracefully with a clear error message if the server rejects.
4. Do NOT attempt to fix the client-time vs server-time issue by adding server-side time to the client hook -- the server is the authoritative check. The client is just a UX hint.

### F-005: Use dynamic currency symbol from refund response

1. In `refund-dialog.tsx:29`, replace the hardcoded `` ₽ `` with a lookup from the response's `result.currency` field.
2. Import or inline a currency symbol map (one already exists in `receipt-scan-dialog.tsx` as `CURRENCY_SYMBOLS`). Extract it to a shared utility.
3. Change the toast to: `` `Vozvrat ${result.amount} ${getCurrencySymbol(result.currency)} oformlen` ``.

### F-006: Handle one-time scan period correctly

1. In `use-payment-form.ts:80`, instead of mapping `'one-time'` to `'monthly'`, either:
   - (A) Map it to a special `'custom'` period with `period_days` set to 0 or null, indicating a non-recurring payment, OR
   - (B) Keep the `period` field as-is and show a prominent notice in the form: "This was detected as a one-time payment. Choose a recurring period if this is actually recurring."
2. Option B is preferred because the user should explicitly decide. Pre-fill `period` as `'monthly'` only if `scanData.period` is undefined/null.
3. If `period === 'one-time'`, pre-select the period dropdown to a "One-time" option if the schema supports it, OR leave it blank and require the user to choose.
4. Check the payment form schema (`paymentFormSchema`) to see if it supports a one-time period. If not, this may require adding it to the schema and the backend.

### F-007: Track refund completion and prevent dialog reopen

1. In `refund-dialog.tsx`, after successful refund (`step === 'success'`), set a ref or state `refundCompleted = true`.
2. When `onOpenChange` fires (dialog close), if `refundCompleted`, do NOT reset step to `'initial'` -- keep it at `'success'` or disable the trigger button entirely.
3. Better approach: after successful refund, `useIsRefundEligible()` will return `false` on next render (because `refundedPayments.length > 0`), which hides the `RefundDialog` button in `settings/billing/index.tsx:161`. Ensure query invalidation triggers a re-render that hides the button.
4. Add `disabled` state to the refund button while the mutation is in-flight to prevent double-clicks during processing.

## Research Sources

- YooKassa Idempotency Key documentation: backend already uses deterministic SHA-256 keys with hour-bucket granularity, confirming that within the same hour, duplicate payments are safe. The risk is cross-hour or plan-change retries.
- Backend `request-refund/index.ts:53` uses `daysSince > 7` (strict greater-than), while client uses `<= 7` (less-than-or-equal). This creates a 1-second window at exactly the 7-day boundary where client shows the button but server rejects.
- The `cancel-subscription` edge function supports `cancel`, `pause`, `reactivate` but NOT `discount` -- confirming F-003 that the discount acceptance is entirely client-side theater.

/**
 * Module: use-checkout
 * Role: API-only checkout hook -- creates payment, returns confirmation token
 * Uses: use-subscription (useInitiateCheckout mutation)
 * Used by: upgrade-dialog
 *
 * CHANGED in BUG-143: Removed widget creation/render (moved to useYooKassaWidget).
 * startCheckout() renamed to createCheckout(). Returns { confirmationToken } or null.
 *
 * CHANGED in BUG-107 F-001: Added paymentId tracking and re-entry guard.
 * createCheckout() is idempotent while paymentId is set. resetCheckout() clears state.
 */

import { useCallback, useRef, useState } from 'react'
import { useInitiateCheckout } from '@/hooks/api/use-subscription'

interface CreateCheckoutResult {
  confirmationToken: string
}

export function useCheckout() {
  const [isLoading, setIsLoading] = useState(false)
  const [error, setError] = useState<string | null>(null)
  const [paymentId, setPaymentId] = useState<string | null>(null)
  const confirmationTokenRef = useRef<string | null>(null)
  const createPayment = useInitiateCheckout()

  const createCheckout = useCallback(
    async (
      plan: 'pro_monthly' | 'pro_yearly'
    ): Promise<CreateCheckoutResult | null> => {
      // F-001 guard: if a payment already exists, return cached token
      if (paymentId && confirmationTokenRef.current) {
        return { confirmationToken: confirmationTokenRef.current }
      }

      setIsLoading(true)
      setError(null)

      try {
        const { confirmation_token, payment_id } =
          await createPayment.mutateAsync(plan)
        setPaymentId(payment_id)
        confirmationTokenRef.current = confirmation_token
        return { confirmationToken: confirmation_token }
      } catch (err) {
        setError(
          err instanceof Error ? err.message : 'Error creating payment'
        )
        return null
      } finally {
        setIsLoading(false)
      }
    },
    [createPayment, paymentId]
  )

  const resetCheckout = useCallback(() => {
    setPaymentId(null)
    confirmationTokenRef.current = null
    setError(null)
  }, [])

  return {
    createCheckout,
    resetCheckout,
    paymentId,
    isLoading: isLoading || createPayment.isPending,
    error,
  }
}
