# Гипотезы: Persona Diversity — что тестировать дальше

**Date:** 2026-02-13
**Context:** После Experiment #5 (stout run 1=27, run 2=60 defects)
**Baseline:** 6 parallel Explore agents × stout prompt = 60 unique bugs за $6-10

---

## H1: User prompt vs Skill prompt

**Наблюдение:** Стаут сработал из ПОЛЬЗОВАТЕЛЬСКОГО промпта, а не из закодированного скилла. Агент получил персону от человека, не от системного промпта.

**Вопрос:** Имеет ли значение ИСТОЧНИК промпта? Если ту же персону прописать в skill/agent prompt — будет тот же эффект или деградация?

**Почему может быть разница:**
- User prompt = "задание от босса" (приоритет в attention)
- System prompt = "должностная инструкция" (фоновый контекст)
- Модель может по-разному взвешивать source of instruction

**Тест:** Тот же stout prompt, но как system prompt в skill. Сравнить количество и качество findings.

---

## H2: Недетерминированность промпта (многоточия как фича)

**Наблюдение:** Промпт был намеренно НЕдетализированный. "Яростный критик... порви проект... стаут за баг." С многоточиями. Без конкретных инструкций ЧТО искать и КАК.

**Вопрос:** Вагусность = фича? Агенты ДОДУМЫВАЮТ свою роль, и каждый додумывает ПО-СВОЕМУ → естественная диверсификация без явного программирования?

**Два конкурирующих эффекта:**
- (+) Недетерминированность → каждый агент интерпретирует роль уникально → больше покрытие
- (-) Недетерминированность → агент может "додумать" не в ту сторону → шум

**Тест A:** Детализированный промпт ("ищи state management баги, race conditions, missing validation...") vs вагусный ("порви проект"). Сравнить coverage и precision.

**Тест B:** Один и тот же вагусный промпт 6 раз (без разных персон). Если coverage всё равно высокий — дело в вагусности, не в персонах.

---

## H3: Teammates (общение) vs Isolated subagents (параллель)

**Наблюдение:** Stout использовал isolated Explore agents (каждый в своём контексте, не общаются). Real teammates (Agent Teams) нашли UDE-7 через adversarial CHALLENGE (один агент спросил другого).

**Вопрос:** Если 6 персон будут общаться друг с другом (Agent Teams / mailbox) — это УЛУЧШИТ или УХУДШИТ качество?

**Аргументы за улучшение:**
- Cross-pollination: "Критик нашёл X" → Безопасник думает "а что если X = вектор атаки?"
- Adversarial challenge: один агент ломает выводы другого
- Пример: UDE-7 (blank screen) найден ТОЛЬКО через challenge

**Аргументы за ухудшение:**
- Groupthink: агенты начинают соглашаться друг с другом
- Anchoring: прочитал чужой вывод → ищешь подтверждение, а не новое
- Шум: переписка ест контекст, меньше места для analysis
- Experiment #2 доказал: single context = single blind spot

**Тест:** Те же 6 персон на том же баге, но через Agent Teams (teammates). Сравнить: количество уникальных findings, overlap между агентами, наличие "challenge-only" findings.

## H4: Personas + TOC frameworks = больше линз

**Наблюдение:** v4 (архитектурная линза) нашёл Reply keyboard state filters и router priority — stout НЕ нашёл. Разные подходы находят разное.

**Вопрос:** Что если не 6 агентов, а 9-12-15? Добавить к 6 персонам ещё 3-6 framework-агентов?

**Кандидаты для добавления:**
- TOC CRT analyst (ищет root cause chains)
- Pearl Causal analyst (строит causal DAG, проверяет do-calculus)
- Kepner-Tregoe analyst (IS/IS-NOT матрица, evidence first)
- TRIZ analyst (ищет противоречия в архитектуре)
- Systems Thinking analyst (ищет feedback loops, leverage points)
- Adversarial red-teamer (пытается сломать каждый найденный баг-репорт)

**Гипотеза:** Personas активируют РАЗНЫЕ ЗНАНИЯ. Frameworks активируют РАЗНЫЕ МЕТОДЫ. Комбинация даёт покрытие по обоим измерениям.

```
           Persona axis (КТО)
           ┌──────────────────────┐
           │ Critic  Security  UX │
           │ Junior  Architect QA │
Framework  ├──────────────────────┤
axis       │ TOC     Pearl   KT   │
(КАК)      │ TRIZ    Systems Red  │
           └──────────────────────┘
```

**Стоимость:** 12 agents × $1-2 = $12-24 за ран. Rounding error.

**Тест:** 6 personas + 3-6 framework agents на том же BUG-477. Считаем: сколько НОВЫХ багов нашли framework-агенты поверх 60 от stout?

## H5: Персоны в фазах 1-2 (сбор данных), не только в фазе 3 (анализ)

**Наблюдение:** Во ВСЕХ экспериментах (v1-v5) фазы 1-2 spark debug mode были ОДИНАКОВЫЕ. Стандартный reproduce + isolate собирал один и тот же набор файлов. Менялась только фаза 3 (анализ).

**Методологическое значение:** Это делает эксперименты чистыми — одни вводные, разный анализ. Разница в результатах = чисто от линзы.

**Вопрос:** А что если персону дать уже на фазе 1-2? Критик полезет в ДРУГИЕ файлы. Безопасник начнёт с auth/permissions. UX-дизайнер пойдёт по user flow. Сейчас все 6 персон анализировали один и тот же набор кода.

**Гипотеза:** Персона в фазе сбора данных → другие файлы в контексте → другие баги. Diversity не только в АНАЛИЗЕ, но и в НАБЛЮДЕНИИ.

**Аналогия:** 6 детективов получили одни и те же улики (фазы 1-2 фиксированы) и сделали разные выводы. А что если 6 детективов сами собирают улики? Каждый пойдёт в разные места.

**Тест:** Те же 6 персон, но персона задаётся ДО фаз 1-2 (весь spark run под персоной). Сравнить: какие ФАЙЛЫ читал каждый агент + какие баги нашёл.

**Риск:** Персона в фазе 1-2 может СУЗИТЬ scope (безопасник читает только auth, пропускает UX-баги). Текущий подход (стандартный сбор + diverse анализ) может быть оптимальнее.

## Meta: Что мы НЕ тестируем (пока)

- Другой проект (generalizability) — нужен, но сначала разобраться с механизмом
- Другие модели (Gemini, GPT, DeepSeek) — отдельный research direction
- Промпт-инжиниринг персон (какие слова лучше) — преждевременная оптимизация
- Оптимальное количество агентов — сначала понять работает ли вообще >6
