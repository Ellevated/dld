```
{project_id}:{spec_id}             # autopilot: e.g. "dld:FTR-146"
{project_id}:inbox-{YYYYMMDD-HHMMSS}  # inbox tasks: e.g. "dld:inbox-20260318-143022"
{project_id}:qa-{task_label}       # QA dispatch: e.g. "dld:qa-FTR-146"
{project_id}:qa-inbox-{...}        # QA for inbox (secondary, no notify)
{project_id}:reflect-{task_label}  # Reflect: e.g. "dld:reflect-FTR-146"
```

- `claude-runner` — Claude tasks (2 parallel slots)
- `codex-runner` — Codex tasks (1 slot)
- `gemini-runner` — Gemini tasks (1 slot)
- `night-reviewer` — Night review (1 slot, separate group)

## Key Architectural Findings

### 1. qa-loop.sh is a zombie (important!)

`qa-loop.sh` is listed in dependencies.md as used by `orchestrator.sh:267`, but in the **actual orchestrator.sh code**, `dispatch_qa()` does NOT call `qa-loop.sh`. Instead, the comment says "Callback owns QA/Reflect dispatch." The QA dispatch actually happens in `pueue-callback.sh:386` via `run-agent.sh`. `qa-loop.sh` still exists and still uses `db_exec.sh` with direct raw SQL, but it may not be called in the current flow. Verify before deleting.

### 2. Two parallel QA paths

Path A: `pueue-callback.sh` → `pueue add ... run-agent.sh ... /qa {spec_id}` (current live path)
Path B: `qa-loop.sh` → `claude --print /qa {spec_id}` directly (bypasses Agent SDK, bypasses run-agent.sh)

These produce different output formats and different phase transitions. New Python orchestrator must unify to Path A only.

### 3. db_exec.sh is legacy technical debt

Used only by `qa-loop.sh` for direct SQL. All other components use `python3 db.py`. The `db_exec.sh` direct SQL approach is the anti-pattern (ADR-017). Delete with qa-loop.sh.

### 4. notify.py has spec parser embedded (unexpected)

`notify.py:134` contains `_parse_spec_for_approval()` — a 100-line spec file parser used by Telegram auto-approval messages. This is Telegram-specific business logic inside the notify module. New orchestrator: keep notify.py intact, import `send_to_project()` from it.

### 5. CLAUDE_CURRENT_SPEC_PATH contract

`inbox-processor.sh` exports `CLAUDE_CURRENT_SPEC_PATH` as env var pointing to the moved `done/` file. The spark skill reads this to write back `**SpecID:** {TASK_ID}`. The callback then reads this spec_id for QA dispatch. This 3-hop contract must be preserved in the new Python orchestrator.

### 6. .run-now-{project_id} trigger files

`telegram-bot.py` creates these files when user sends `/run {project_id}`. The orchestrator checks for them each cycle to trigger immediate processing. New Python daemon must watch for these (inotify or poll).

## Risks

1. **Risk:** pueue callback interface is fixed (pueue.yml config)
   **Impact:** New Python callback must accept same CLI args: `<id> '<group>' '<result>'`
   **Mitigation:** Keep callback as a thin Python script with same CLI signature; pueue.yml doesn't change

2. **Risk:** Telegram bot (telegram-bot.py) creates `.run-now-{project_id}` trigger files
   **Impact:** New Python orchestrator must still poll for these files
   **Mitigation:** Preserve file-based trigger mechanism; only internals change

3. **Risk:** qa-loop.sh uses direct SQL via db_exec.sh (not parameterized db.py)
   **Impact:** Race condition risk if qa-loop.sh is still being called
   **Mitigation:** Verify qa-loop.sh is dead code; if confirmed, delete cleanly

4. **Risk:** `.claude/skills/spark/completion.md:190` references pueue-callback.sh
   **Impact:** Skill docs become misleading after rewrite
   **Mitigation:** Update spark/completion.md to reference new Python callback

5. **Risk:** CLAUDE_CURRENT_SPEC_PATH env var handoff (inbox → agent → callback)
   **Impact:** If new orchestrator doesn't set this env var, QA dispatch loses spec_id
   **Mitigation:** New `claude-runner.py` already passes it via `ClaudeAgentOptions.env`; orchestrator must set it before pueue add

6. **Risk:** No tests exist for any orchestrator script
   **Impact:** Rewrite regression risk is high — no safety net
   **Mitigation:** Write integration tests alongside rewrite (real pueue + real sqlite, per ADR-013)

7. **Risk:** setup-vps.sh generates systemd unit with hardcoded ExecStart paths
   **Impact:** After rewrite, new Python entry point must replace orchestrator.sh in ExecStart
   **Mitigation:** Update setup-vps.sh template; existing deployments need manual systemd reload
